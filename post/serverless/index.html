why serverless

November 29, 2016

Focus on your strengths, your business.

This year, 2016, I’m getting married (woo!)

Why is that relavent? Because with my new AWS Solutions Architect Cert under my belt, I have the perfect excuse to build the worlds most robust wedding website ever.

But first lets try to define “Serverless,” this guy sums it up perfectly:

“Serverless architectures refer to applications that significantly depend on third-party services…. such architectures remove the need for the traditional ‘always on’ server system sitting behind an application. Depending on the circumstances, such systems can significantly reduce operational cost and complexity” - Martin Fowler

Thanks, Martin.

Content

I’ve worked with Content Management Systems before; big expensive ones, free open-source, and others in-between. But they all suffer from similar problems, they:

  1. Don’t do version control well
  2. Don’t scale well
  3. Require a server, which
    • Need maintenance
    • Cost money

But mostly, I’m just over maintaining servers!

Lets address those problems:

Version Control

I’m not sure this is even up for discussion - if you need source version control, use Git.

Moving on.

Durability / Availabily / Security

Let me lay down some quick perks of running my serverless wedding website:

  • Guarenteed uptime of 99.99%
  • Guarenteed durability of 99.999999999%
  • Protection against DDOS
  • Global CDN caching and SSL offloading
  • No maintenance

Let’s not kid ourselves, there are still servers involved in serverless architectures; this is not wizardry. In reality it’s simple; somebody deals with the many invisible layers which enables this service, including server maintenance, then charge a small fee to consume a small slice of it. Fortunately for me, AWS is doing this at such a massive scale that my small website is a drop in the ocean and priced accordingly.

Money

Have a think, what would you expect to pay for the above features with a traditional server architecture?

Even if you wanted to use virtualization on the cheapest cloud provider out there, achieving those SLA’s would cost thousands.

I did in under AUD 20, here’s the breakdown:

Up-front costs - AUD 12.86

  • Domain name - $1.99
  • SSL Certificate - $10.87

(yes, there are cheaper options out there if you’re less picky)

Monthly costs - Assuming AWS Free-Tier - AUD 1.45

  • Domain Name Management - $0.50
  • DNS requests - $0.01
  • Web Hosting (S3) - Free
  • Presentation Layer (API Gateway) - Free
  • Application layer (Lambda) - Free
  • Database (DynamoDB) - Free
  • SMS Notifications - $0.94

Monthly costs without Free-Tier

Rough estimates say about AUD 2.00, but for the sake of argument I calculaed for about a million hits a day, including database writes, doesn’t even reach AUD 100.

Scale

Here’s the cherry on top: Lets say I did want to scale my website from 100 monthly users to 100 million - no extra effort is required on my behalf, just a bigger budget. And lets say I want to prevent a burst of traffic from blowing my budget, there’s tools for that too.

My Serverless Solution

Okay, so what does it look like? I chose AWS, so here’s my very AWS-centric architecture:

  • Git for source / version control / collaboration
  • Hugo provides a great way to structure the site and render markdown to HTML
  • Static pages are pushed to S3 for hosting
  • CloudFront provides a CDN and offloads SSL encryption
  • JavaScript + Vue.js for front end application logic
  • API Gateway and Lambda for backend logic
  • DynamoDB for data storage
  • SNS for notifications
